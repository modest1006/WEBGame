# WEBGame 実装規約

このリポジトリでゲームを作る・直すときは、以下の5原則を必ず守る。

## 1. ファイル構成を分ける（単一巨大HTML禁止）

1ファイルにHTML/CSS/JSを詰め込まない。役割ごとにモジュール分割する。

```
games/<game-name>/
├── index.html          # マークアップのみ（インラインscript/styleは最小限）
├── README.md           # 遊び方・操作方法・起動方法・デバッグAPI一覧
├── css/
│   └── style.css       # そのゲーム専用のデザイン
└── js/                 # 通常スクリプト（index.htmlで依存順に読み込み）
    ├── main.js         # エントリポイント・ゲームループ
    ├── game.js         # ゲームロジック（DOM/Canvas非依存に保つ）
    ├── renderer.js     # 描画
    ├── input.js        # 入力（キーボード＋タッチ）
    ├── audio.js        # 効果音（WebAudio）
    └── debug.js        # デバッグ機能（必須、下記参照）
```

- **ロジックと描画を分離する。** `game.js` はDOMもCanvasも触らない純粋なステートマシンにする。これによりデバッグAPIから時間を進めて（`step(ms)`）ヘッドレスに検証できる。
- **`index.html` のダブルクリック（`file://`）だけで遊べるようにする。** ES modules（`import/export`）は `file://` でCORSブロックされるため使わない。通常の `<script src>` を依存順に並べて読み込む（トップレベルの `const`/`class` はスクリプト間で共有される）。同じ理由で `fetch` によるローカルファイル読み込みも避ける。
- セルフレビュー時のブラウザ操作にはローカルサーバーを使う（`.claude/launch.json` に登録して `preview_start` で起動）。

## 2. モデル自身でデバッグできる機能を必ず実装する（セルフレビュー必須）

実装したら「動くはず」で終わらせない。**自分でプレイして検証できる仕組みをゲーム側に埋め込む**こと。

必須のデバッグ機能（`js/debug.js`）:

- `window.__game` にデバッグAPIを公開する。最低限:
  - `getState()` — スコア・状態・盤面などをシリアライズ可能なスナップショットで返す
  - `dump()` — 盤面/シーンをASCIIアートで返す（`preview_eval` で目視確認するため）
  - `step(ms)` — ポーズ中でも時間を指定分だけ決定論的に進める
  - ゲーム固有の状態注入（例: 盤面セット、任意ピース出現、レベル変更、即ゲームオーバー）
- `?debug=1` またはバッククォートキーでデバッグオーバーレイ表示（FPS・内部状態）
- `?seed=<n>` で乱数を固定し、再現可能なテストを可能にする

- **ステージ/レベルをコード定義するゲームは、全ステージの「クリア可能性」を機械的に検証する**。①グリッドならBFSで到達可能性チェック（穴・奈落は不通扱い） ②可能なら `__game` APIでウェイポイント追従の自動操縦を書き、物理込みで実走破させる（ROLL MAZEで「唯一の経路が穴で塞がれた詰みステージ」を出荷後にユーザーが発見した反省）

- **グリッドの「全体シフト」は座標系の整合を必ず確認する。ヘックス（オフセット）格子で1行シフトは剛体移動にならずせん断する**。偶数/奇数行でセルのx座標が半セルずれる格子では、`(row,col)→(row+1,col)` と列を据え置きで1行下へ写すと、行ごとにx方向へ逆向きにずれて配置が歪み、隣接（連結）関係まで静かに変わる（BUBBLE EXの「3つ繋がっても消えない」「片色を消すと全落下」の真因）。**盤面を動かしたいときはグリッドをリマップせず、`offsetRows`のような量を物理・描画・判定に一律に足すオフセット方式**にする（幾何と連結性が完全保存）。剛体移動が必要なら偶数行ぶん（2行単位）シフトなら成立する。連結/消滅系は「見た目の密着」でなく必ず**グリッド隣接トポロジー**で判定し、`GH.neighbors` 等で数千回ファズ（見逃しpop0・浮き玉0・孤立残留0）して確認する

**セルフレビュー手順**（実装完了後に必ず実行）:

1. `preview_start` でサーバー起動
2. `preview_console_logs` でエラーゼロを確認
3. `preview_eval` で `__game` APIを叩き、主要ルールを検証する（例: テトリスならライン消去・スコア計算・ゲームオーバー条件を状態注入して確認）
4. `preview_screenshot` / `preview_snapshot` で見た目とレイアウト確認（モバイルは `preview_resize` の mobile プリセットでも確認）
5. デバッグAPIの一覧はゲームの README に記載する

## 3. ディテールにこだわる（ジュース感）

「動くだけ」のゲームにしない。以下を意識して実装する:

- **アニメーション/モーション**: イージング、消滅・出現演出、パーティクル、画面シェイク、フラッシュなどのフィードバック
- **ディシジョン（手触り）**: 入力の応答性（DAS/ARRのようなキーリピート設計、先行入力、猶予フレーム）、ゴースト表示などプレイを快適にする定番仕様
- **効果音**: WebAudioのオシレータで軽量に自前合成でよい。外部音源ファイル不要。ミュートボタンを付ける
- 演出はゲームロジックからイベント発火（`onEvent(type, data)`）で疎結合に
- **エフェクトはバリエーションを出す。** 全イベントに同じパーティクルを使い回さない。エンティティの種別・サイズ・重要度ごとに色/形状/量/シェイク強度を変える（例: 雑魚撃破=小爆発、ボス=多段爆発＋スロー、レア進化=専用演出）。同一演出でも乱数で角度・寿命・サイズを散らす
- 「状態の高まり」を複数チャネルで同時に表現する（例: コンボ上昇→BGMレイヤー追加＋UIパルス＋オーラ色変化）。1チャネルだけだと気づかれない

## 4. デザインは雰囲気に合わせ、UIはゲーム毎に専用デザイン

- 共通UIテンプレートを使い回さない。ゲームの世界観（レトロアーケード、ネオン、和風、ミニマル等）を決めてから配色・フォント・エフェクトを設計する
- タイトル画面・ポーズ・ゲームオーバー・スコア表示まで含めて世界観を統一する
- フォントは外部CDN依存でもよいがフォールバックを必ず指定する

## 5. PC操作とスマホ操作の両対応

- **キーボード/マウス**と**タッチ**の両方を必ず実装する
- タッチは「画面上ボタン」と「ジェスチャー（スワイプ/タップ）」を適材適所で。`pointer: coarse` メディアクエリでタッチUIの表示を切り替える
- `touch-action: none` や `preventDefault` でスクロール/ダブルタップズームの誤爆を防ぐ
- viewport メタタグ必須。モバイル縦画面レイアウトを `preview_resize`（mobile）で必ず確認する

## ジャイロ操作（DeviceOrientation）

- **スマホ縦持ち（beta≈60〜90°）はオイラー角Z-X-Yのジンバルロック帯。`beta/gamma のニュートラル差分`で入力を作ると、gammaが±180フリップ・過大値で振り切れ「曲がりが直角」「ピッチが効かない」になる**（PROPELLA実機で発生）。→ (alpha,beta,gamma)からZ-X-Y intrinsic順でクォータニオンを構築し、キャリブレーション姿勢との**相対回転**から軸を導出する: pitch=画面法線の仰角差分、turn=画面法線を水平面に射影した方位角差分（=鉛直軸まわりの「ひねり」。ユーザーが直感的なのはバンクでなくこのひねり）。方位角差は必ず±180正規化
- 感度はデッドゾーン（2〜3°）＋フルスケール（25〜35°）＋**expoカーブ（sign(x)·x²）**で中央を繊細に。ローパス（`1-exp(-dt*8)`程度）でジッタ除去
- iOS 13+は `DeviceOrientationEvent.requestPermission()` をユーザータップから呼ぶ許可フロー必須。HTTPSのみ動作（file://・PCにはキーボード/マウス/ドラッグのfallbackを必ず用意）
- 検証は合成 `deviceorientation` イベント（`new Event`にalpha/beta/gammaを付与してdispatch）でキャリブレーション→入力変換をE2Eで通す。ジンバル帯（beta=85〜90）に±2°ノイズを入れて入力が暴れないことも確認

## Canvas 2D のパフォーマンス

- **エンティティ単位の `shadowBlur` / radial gradient 描画は禁止**（数百個で桁違いに遅くなる。PETRIで100フレーム2分超の実例）。グローは「低解像度オフスクリーンに描く→拡大して `lighter` で1回合成」の定数コスト方式で
- セル/タイル状の大量描画は `ImageData`＋`putImageData`→`imageSmoothingEnabled=false` 拡大
- 静的な背景・枠・ビネットはオフスクリーンに1回描いてキャッシュ、毎フレームは drawImage 1回
- 更新頻度の分離: 盤面が変わった時だけ重い層を再構築し、rAF毎は軽いオーバーレイのみ
- タッチ端末は devicePixelRatio を1.0〜1.5に制限

## 3Dゲーム（Three.js）

- `file://` 直開き規約と両立させるため、**Three.js r147のUMDビルド（three.min.js）をゲーム内 `lib/` に同梱**して通常scriptで読む（グローバル`THREE`が生える）。r148以降はESM専用なので使わない。CDN参照はオフラインで死ぬため不可
- ロジック（物理・判定）は従来どおり `game.js` にThree非依存で実装し、renderer.jsだけがTHREEを触る。これで `__game.step()` によるヘッドレス検証がそのまま効く
- `WebGLRenderer` は `preserveDrawingBuffer: true` で生成する（ヘッドレス検証の `canvas.toDataURL` に必須）
- **カメラのnear/far・フォグのnear/farは必ずシーンスケール（カメラ距離）から導出する**。固定値だと「シーンはあるのに真っ黒」事故になる（BUBBLE EXで far=200/フォグ260 vs カメラ距離855 の全没が実際に起きた）。「draw callsは出ているのに画面が背景色一色」はフォグ沈没を第一に疑う
- **同じ座標変換を使うオブジェクトは同じ親Groupのオフセット下に置く**。盤面を `boardGroup`（原点合わせのpositionオフセット付き）に入れているのに、パーティクル/破片/落下ゴーストを別の無オフセットGroupに入れると、同じ `toScene()` で計算しても `(+W/2, -H/2)` ぶんズレて湧く。デスクトップでは画面内でも、モバイル縦だと画面外に飛んで「演出が出ない」に化ける（BUBBLE EX v3で実際に発生）。→ 演出用Groupは `particleGroup.position.copy(boardGroup.position)` で必ず揃える
- **背景を「全オブジェクトに対して確実に最背面」にするなら二段描画**。大きく傾けた床plane等は、手前側の辺がプレイ面(Z=0)より前（正のZ）に飛び出して前景の玉を隠すことがある（BUBBLE EX v3: 700四方・-69°の床の near辺がZ≈+167まで前進し最下部の玉を occlude）。Zの調整で対処すると脆いので、`autoClear=false`＋パス1で背景のみ描画→`clearDepth()`→パス2で前景を描く二段構成にすると、Z重なりに関係なく背景が必ず後ろになる（`group.visible` トグルで1シーンのまま実現可）
- モバイルは `pixelRatio` 上限1.5・影テクスチャ1024以下に抑える
- **キャラクター/クリーチャーをプリミティブの素組み（円錐＋球等）で作ると「記号的でしょぼい」とユーザーに却下される**（GHOST LENSで実例）。外部アセット禁止でも表現手段はある: ①**プロシージャルCanvasTexture**（2D canvasにradialGradientの陰影・顔・布・髪・模様を描いてテクスチャ化。起動時1回生成、毎フレーム再描画禁止）②ShaderMaterialの頂点変位（布の揺らぎ）とフレネル縁発光 ③点群パーティクルの形状集散。特に「顔」はcanvas描画テクスチャが最も効く。造形後は必ず正面キャプチャで「何の姿か一目で分かるか」を確認（側面図で組むと正面から全翼機/UFO化する事故が起きる。人型なら 幅:高さ≤0.6・頭:胴=1:2.5〜3.5 を目安に投影サイズで自己検査）。多関節の黒シルエットは正面が幾何学的に化けやすいので、常に3/4視になるようyawオフセットを入れると安定

## セルフレビューの必須経路

- **音声をunlockした状態でも一通り動かすこと。** `music.unlock()` / `sfx.unlock()` を呼ばないテストは全SFXコードが早期returnでスキップされ、SFX内のバグ（未定義メソッド等）を見逃す。AudioContextはユーザー操作なしでも生成でき（suspendedのまま）、コード経路の検証には十分
- ゲームループ（rAF）とイベントリスナーには例外境界を置く（try/catch＋console.error）。1つの演出バグで全体がフリーズする事故を防ぐ

## デプロイ（GitHub Pages）

- Pagesは全ファイル `max-age=600` でキャッシュされるため、**デプロイ直後は新旧ファイル混在が起きる**。JS/CSSを変更したら index.html のアセット参照のバージョンクエリ（`?v=N`）を必ず上げること
- 「反映されない」報告はまずキャッシュを疑う（マーカー文字列を `Invoke-WebRequest` で確認する）
- **`?v=N` バンプにPowerShellの `Get-Content -Raw | -replace | Set-Content` を使わない**。BOMなしUTF-8の日本語ファイルをShift-JISで読んで文字化けさせ、そのままコミット・配信する事故が実際に起きた（petri/rollmazeで発生、grep '[ぁ-ん]' の消失で検出）。バンプは**Editツールの replace_all** で行うこと

## 起動登録・周辺更新

新ゲーム追加時は以下も併せて更新する:
- `.claude/launch.json` に静的サーバー設定を追加（Python の `http.server` に `--directory games/<name>`。ポートはゲーム毎に変える）
- ルートの `index.html`（アーケードポータル）にカードを追加
- **ルートの `README.md` のゲーム一覧に行を追加**（ジャンル・概要1行）
